Why The Toolbox Matters

`.dev-tools` is not just a place to stash scripts. It is a governed build surface with explicit boundaries between what stays in the toolbox, what gets copied into target projects, and what remains documented as reusable project doctrine.

Portable Agent-first Project-agnostic

The Problem It Solves

Without a stable toolbox, each new project improvises its own helper scripts, docs, and onboarding rituals.
Without explicit boundaries, helper tooling quietly leaks into runtime dependencies or drags project-specific assumptions along with it.
Without shared doctrine, humans and agents end up solving the same orientation problems over and over.

What This Repo Establishes Instead

One Toolbox, Many Projects

The builder tools stay here and can work on many projects without being rewritten each time.

Vendable Capability

Packages and templates are copied into target projects only when they are actually needed.

Shared Build Language

The README, contract, manifests, and workflow docs give humans and agents the same orientation surfaces.
