The System in One Pass

1. Orient Cleanly

A good session starts from the manifests and doctrine surfaces, not from random file wandering. Humans get the walkthrough and README; agents get toolbox_manifest.json, tool_manifest.json, and the agent guide.

2. Work Inside Boundaries

Builder tools stay in the toolbox, vendable packages move into projects, and sidecar/container surfaces keep local-agent work inspectable without blurring trust boundaries.

3. Preserve Continuity

The experiential workflow, dev log, manifests, and journal-oriented doctrine keep work legible across pauses, handoffs, and different agents.

4. Vend Without Drift

When a target project needs the toolbox, install the full sidecar or vend the specific package or template it actually needs. The target project gets what it needs without turning this repo into an undeclared runtime dependency.

How To Enter The Workflow

Start here: use this walkthrough as the human-readable front door into the toolbox.
Read the core doctrine: move into README.md, CONTRACT.md, VENDORING.md, and _docs/EXPERIENTIAL_WORKFLOW.md.
Choose the right surface: decide whether the task belongs in a builder tool, packed authority, vendable package, or vendable document.
Go operational only after orientation: use install.py, python src/tools/sidecar_install.py, python src/mcp_server.py, or package-level smoke tests once the intent is clear.
The point of this microsite is not to replace the raw docs. It is to make the system understandable enough that the raw docs feel welcoming instead of overwhelming.

How The Human Fits Into The Loop

The best version of this toolbox is not “agent runs alone.” It is a tight human-builder loop where the person provides intent and judgment, and the doctrine keeps the work legible and resumable.

1. Human names the need: what matters now, what success looks like, and what should stay out of scope.
2. Builder names the method: which surface fits, what will change, and how it will be verified.
3. Human reviews the evidence: smoke tests, doc updates, outputs, or architecture movement.
4. Builder leaves a clean return path: update the dev log and relevant doctrine so the next session can resume without guesswork.

The Current Local-Agent Runway

Tranche 9: Local Sidecar Agent Runtime using Ollama and only the guarded toolbox surface.

The Parking Loop

The toolbox stays teachable when meaningful work ends with a clean return path. Before leaving, the builder should:

record the architectural truth in the docs that future sessions will actually read
update _docs/DEV_LOG.md when the toolbox itself changed in a meaningful way
keep manifests and onboarding surfaces aligned with the real repo shape
leave the next human or agent a state they can inspect without reconstructing intent from diffs alone

In short: orient -> choose -> act -> verify -> resync.
